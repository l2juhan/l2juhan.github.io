##  소프트웨어 테스팅 개요 및 블랙 박스 테스팅 기법 정리

###   1. 테스팅 정의 및 목적

#### 테스팅이란?

소프트웨어에 숨어있는 결함(Fault)을 찾기 위해 프로그램을 실행하는 일련의 행위와 절차

IEEE(1993) 정의에 따르면, 시스템이 정해진 요구를 만족하는지, 예상과 실제 결과의 차이를 수동/자동 방법으로 검사하고 평가하는 과정

#### 테스팅의 목적

프로그램의 잠재된 오류 발견, 기술적 기능 및 성능 확인, 사용자 요구 만족도 향상, 제품 신뢰도 향상이 주요 목적

중요한 점은 **결함이 없음을 증명하는 것이 아니라 결함이 존재함을 보여주는 작업**이라는 것

Dijkstra의 명언: "Testing shows the presence, not the absence of bugs."

#### 테스팅의 한계

완벽한 테스트는 불가능→이유는 프로그램 내부 조건의 무수한 조합, 입력값의 무한한 조합, GUI 이벤트 순서의 무한한 조합 때문

#### 테스팅의 이점

- **디버깅 감소**: 테스트 거친 코드는 결함이 적어 디버깅 시간 절약

- **자신 있게 변경 가능**: 리팩터링 작업이 쉬워짐

- **더 나은 문서자료**: 테스트 코드가 동작 명세 역할 수행

- **더 단순한 리뷰**: 리뷰어의 검증 시간 절약

- **사려 깊은 설계**: 모듈화가 잘 된 설계 유도

- **고품질 릴리즈를 빠르게**: 자동 테스트로 불안감 해소

---

###   2. 테스트 작업 과정

#### 테스트의 단계

1. **목표설정**: 테스트로 점검할 항목 정의 (기능 완벽성, 신뢰도 등)

1. **방법 결정**: 검사, 증명, 블랙박스/화이트박스 테스트, 자동화 도구 선택

1. **테스트 케이스 선택/작성**: 테스트 자료, 시행 조건 개발

1. **테스트 오라클 작성**: 예상되는 올바른 결과 정의

1. **테스트 실행**: 테스트 하니스(test harness) 활용

1. **실행 결과 분석**

###   3. 테스팅 종류

#### 프로그램 실행 여부에 따른 분류

| 분류 | 설명 | 예시 |
| --- | --- | --- |
| 동적 테스트 | 프로그램 실행을 요구하는 테스트 | 화이트박스, 블랙박스, 명세/구조/경험기반 |
| 정적 테스트 | 프로그램 실행 없이 구조 분석 | 코드 검사, 워크스루(Walkthrough) |

#### 테스트 목적에 따른 분류

| 테스트 종류 | 설명 |
| --- | --- |
| 회복(Recovery) 테스트 | 실패 유도 후 회복이 적절히 수행되는지 검증. OS, DBMS, 통신용 SW 안정성 테스트에 중요 |
| 안전(Security) 테스트 | 불법적 침투로부터 시스템 보호 기능 검증. 기밀성, 무결성, 사용자 인증, 접근 통제, 부인봉쇄 등 테스트 |
| 강도(Stress) 테스트 | 비정상적인 값/양/빈도 입력에 대한 정상 수행 상태 테스트. 민감도 테스트 포함 |
| 구조(Structural) 테스트 | 내부 논리적 경로의 복잡도 평가. 화이트박스/유닛 테스트로도 분류 |
| 성능(Performance) 테스트 | 실행 시간, 자원 이용률, 처리 시간, 응답 시간 등 목표치 달성 여부 테스트 |

#### 테스트 단계에 따른 분류 (V 모델)

V 모델은 요구사항 정의 → 분석 → 설계 → 구현 단계와 테스트 과정이 V자로 연결됨

추가로 **회귀 테스트**도 포함됨

                                                           │

│ 요구분석   │◄ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ►│ 시스템 테스트 │◄───┘

     │                                         │

└──────┬───────┘                  └──────┬──────┘
       │                                │
       ▼                               │
  ┌──────────┐       모듈 검증    ┌─────────────┐
  │ 모듈 설계 │◄ ─ ─ ─ ─ ─ ─  ─ ─►│  단위 테스트  │

###   4. 블랙 박스 테스팅

#### 개념

소프트웨어의 특정 기능이 완전히 작동되는 것을 입증하는 테스트(**기능 테스트**라고도 함)

내부 구조를 모르는 상태에서 입력과 출력만 확인함

#### 특징

- 사용자 요구사항 명세 기반 테스트

- 소프트웨어 인터페이스에서 실시

- 테스트 과정의 **후반부**에 주로 적용

#### 발견 가능한 오류 유형

- 부정확하거나 누락된 기능

- 인터페이스 오류

- 자료 구조나 외부 DB 접근 오류

- 행위/성능 오류

- 초기화/종료 오류

---

###   5. 블랙 박스 테스팅 기법들

#### 5.1 동등(동치) 분할 기법 (Equivalence Partitioning)

**개념**: 입력 데이터를 그룹화하여 각 그룹에서 대표값을 선정해 테스트하는 기법

**특징**:

- 입력 데이터에 따른 결과값 테스트

- 단순하고 이해하기 쉬움

- 사용자가 작성 가능

**예시**: 성적에 따른 상여금 계산

| 점수 범위 | 상여금 |
| --- | --- |
| 90~100점 | 600만원 |
| 80~89점 | 400만원 |
| 70~79점 | 200만원 |
| 0~69점 | 0원 |

테스트 케이스: 각 그룹에서 대표값(60점, 73점, 86점, 94점) 선정하여 테스트

---

#### 5.2 경계값 분석 기법 (Boundary Value Analysis)

**개념**: 입력 조건의 경계값에서 오류 발생 확률이 높다는 점을 이용하여 경계값을 테스트 케이스로 선정하는 기법

**특징**:

- 동등 분할 기법의 보완

- 중간값보다 **경계값**에 집중

**예시**: 오전/오후 도출 기능 (점수에 따른 상여금)

테스트 케이스: -1, 0, 1, 69, 70, 71, 79, 80, 81, 89, 90, 91, 99, 100, 101 등 경계값 및 경계 근처 값

---

#### 5.3 오류 예측 기법 (Error Guessing)

**개념**: 각 기능에 대한 제약조건을 위반함으로써 생길 오류를 추측하여 테스트하는 기법

**특징**:

- 다른 기법으로 찾기 어려운 오류 발견

- 보충적 검사 기법

**예시**: 회원가입 기능에서 입력 데이터 없이 요청 시 에러 발생 여부 확인

---

#### 5.4 원인-결과 그래프 기법 (Cause-Effect Graph)

**개념**: 입력 데이터 간의 관계와 출력에 영향을 미치는 상황을 체계적으로 분석하여 효용성 높은 테스트 케이스를 도출하는 기법

**동등 분할/경계값 분석의 단점 극복**: 입력 환경의 복합성을 고려

#### 작성 단계

1. **프로그램 분할**: 적합한 크기로 분할

1. **원인과 결과 찾기**: 명세서에서 원인(입력 조건)과 결과(출력 조건)를 찾아 일련번호와 같은 식별자 부여

1. **원인-결과 그래프 작성**: 논리 연산자로 연결

1. **제한 조건 표시**: 그래프에 제약 추가

1. **의사결정 테이블 변환**

#### 5.5 기타 기법

#### 상태 전이 테스팅

##  화이트 박스 테스팅 기법들 정리

###   1. 화이트 박스 테스팅 개요

**정의**: 모듈의 원시 코드를 오픈한 상태에서 논리적 경로를 고려하여 테스트 케이스를 설계하는 방법

**특징**: 구조적 테스트, 테스트 과정 초기에 적용, 모듈 내부 작동을 직접 관찰 및 모든 문장을 한 번 이상 실행

#### **기법 분류**

#### **기초 경로 검사(Base path testing)**

- 대표적인 화이트 박스 테스트 기법

- 테스트 케이스 설계자가 절차적 설계의 논리적 복잡성을 측정할 수 있게 해주는 기법

- 테스트 측정 결과는 실행 경로의 기초를 정의하는 데 지침으로 사용

#### 제어 구조 검사(Control structure testing)

- **조건 검사**: 프로그램 모듈 내에 있는 논리적 조건을 활용하는 테스트 케이스 설계 기법

- **루프 검사**: 프로그램의 반복 구조에 초점을 맞춰 실시하는 테스트 케이스 설계 기법

- **데이터 흐름 검사**: 프로그램에서 변수의 정의와 변수 사용의 위치에 초점을 맞춰 실시하는 테스트 케이스 설계 기법

#### 화이트 박스 테스트 기법들

- **문장 검증 기준**: 모든 state를 한번씩 지나게 고려

- **분기 검증 기준**: 분기의 모든 경로를 고려

- **조건 검증 기준**: 분기 안에 모든 조건을 고려

- **분기/조건 검증 기준**: 분기+조건

- **다중 조건 검증 기준**

- **기본 경로 테스트**: 전체적인 path로 복잡도 고려

###   3. 화이트 박스 테스트 기법 상세 설명

#### 3.1 문장 검증 기준 (Statement Coverage)

**정의**: 프로그램 내의 모든 문장이 최소한 한 번은 실행되는 테스트 케이스 선정

#### 1단계: 원시코드 → 제어 흐름 그래프

```c++
foo(int X, int Y, int Z) {
	if (X > 1 && Y == 0) {
		Z = Z / X;
	}
	if (X == 2 || Z > 1) {
		Z = Z + 1;
	}
}
```

![소프트웨어 테스팅 개요](/assets/images/posts/software-testing-overview/software-testing-overview-8.png)

#### 2단계: 가능한 모든 경로 구함

| 번호 | 경로(가,다) | 가능 경로 | 만족 여부 | 이유 |
| --- | --- | --- | --- | --- |
| 1 | T,T | a-c-e | 만족 | (가),(나),(다),(라) 문장을 모두 지나감 |
| 2 | T,F | a-c-d | 불만족 | (라)문장 안 지나감 |
| 3 | F,T | a-b-e | 불만족 | (나)문장 안 지나감 |
| 4 | F,F | a-b-d | 불만족 | (나),(라) 문장 안 지나감 |

#### 3단계: 모든 경로 중 문장 검증 기준을 만족하는 경로 선택

#### 4단계: 선택한 경로에 해당하는 테스트 데이터를 가지고 실행

| 번호 | 경로(가,다) | 테스트 데이터 | 가능 경로 | 출력값 | 만족 여부 |
| --- | --- | --- | --- | --- | --- |
| 1 | (T,T) | X=2, Y=0, Z=3 | a-c-e | 2 | 만족 |

**문제점**:

- `and`를 `or`로 잘못 코딩해도 오류 발견 불가

- `Z>1`을 `Z>0`으로 잘못 코딩해도 오류 발견 불가

---

#### 3.2 분기 검증 기준 (Branch Coverage / Decision Coverage)

**정의**: 모든 분기를 적어도 한 번씩 거쳐가도록 (T와 F 각각 한 번씩) 경로 선정

**적용 규칙**:

- **if문**: 참/거짓 두 경로가 각각 한 번씩 선정

- **switch문**: 모든 case문과 default문 선정

- **for,while문**: 적어도 한 번 루프의 내부가 실행

#### 1단계: 원시코드 → 제어 흐름 그래프

#### 2단계: 가능한 모든 경로 구함

#### 3단계: 모든 경로 중 분기 검증 기준을 만족하는 경로 선택(하나만으로 만족하지 못하는 경우, 경로를 묶어서 진행)

![소프트웨어 테스팅 개요](/assets/images/posts/software-testing-overview/software-testing-overview-9.png)

**문제점 - OR 연산인 경우**:

```text
조건문 (다): X=2 || Z>1

• 경로 1,4 사용 시: Z>1를 Z<1로 코딩 실수해도 오류 발견 못함
• 이유: 둘 중 하나라도 T이면 다른 식이 무엇이든 결과에 영향 없음
```

**문제점 - AND 연산인 경우**:

```text
if (score >= 90 && report >= 90)

┌─────────────┬────────┬──────────────┬───────────────┐
│ 개별조건식A    │ 연산자   │  개별조건식B    │  조건문(전체)   │
│ (score>=90) │        │ (report>=90) │               │
├─────────────┼────────┼──────────────┼───────────────┤
│      T      │        │      T       │       T       │
│      T      │  and   │      F       │       F       │
│      F      │        │      T       │       F       │
│      F      │        │      F       │       F       │
└─────────────┴────────┴──────────────┴───────────────┘

• score < 90일 때, report >= 90을 report < 90로 잘못 코딩해도 발견 못함
• 이유: 둘 중 하나라도 F이면 다른 식이 무엇이든 결과는 F
• 해결 방법: 조건문 내 개별 조건식에 대하여 각각 T/F인 경우를 최소한 한 번씩 수행
```

---

#### 3.3 조건 검증 기준 (Condition Coverage)

**정의**: 조건문 내의 개별 조건식에 대해 각각 T와 F인 경우를 최소한 한 번씩 수행

**테스트 케이스 구성**:

```text
조건문 (가): X > 1 && Y == 0

┌──────────────────────────────────────────────┐
│ (가): (X>1)=T and (Y=0)=T                    │
│ (가): (X>1)=T and (Y=0)=F                    │
│ (가): (X>1)=F and (Y=0)=T                    │
│ (가): (X>1)=F and (Y=0)=F                    │
└──────────────────────────────────────────────┘

가능한 조합:
• (T,T), (F,F)  → 개별 조건식 각각 T,F 만족
• (T,F), (F,T)  → 개별 조건식 각각 T,F 만족

```

#### 3.4 분기/조건 검증 기준 (Branch/Condition Coverage)

**정의**: 조건 검증 기준(개별 조건식)을 모두 만족하면서 분기 검증 기준(전체 조건식)도 만족

**테스트 케이스**:

![소프트웨어 테스팅 개요](/assets/images/posts/software-testing-overview/software-testing-overview-11.png)

**마스크(Mask) 문제**:

```text
┌─────────────────────────────────────────────────────────────────────────┐
│ 마스크란?                                                                 │
│ 어떤 개별 조건식에 의해 다른 개별 조건식의 결과와 상관없이                            │
│ 전체 조건문의 결과가 이미 결정되어지는 것                                         │
├─────────────────────────────────────────────────────────────────────────┤
│ 첫 번째 조건문(AND) 문제:                                                   │
│ • (X>1)가 F이면 (Y=0) 결과와 관계없이 조건문이 F                               │
│ • 결과: (Y=0)에서 오류 발생 시 발견 못함                                       │
├─────────────────────────────────────────────────────────────────────────┤
│ 두 번째 조건문(OR) 문제:                                                    │
│ • (X=2)가 T이면 (Z>1)와 관계없이 조건문이 T                                   │
│ • 결과: (Z>1)에서 오류 발생 시 발견 못함                                       │
└─────────────────────────────────────────────────────────────────────────┘

```

---

#### 3.5 다중 조건 검증 기준 (Multiple Condition Coverage)

**정의**: 마스크 문제까지 해결한 테스트 케이스를 생성하는 기준

**해결 방법**:

```text
AND 연결 시 마스크 문제 해결:
┌─────────────────────────────────────────────────────────────────┐
│ [문제] 1번째 조건식이 F인 경우, 2번째가 T든 F든 결과가 F        │
│ [해결] 2번째 조건식에서 T와 F인 경우를 각각 하나씩 추가         │
└─────────────────────────────────────────────────────────────────┘

OR 연결 시 마스크 문제 해결:
┌─────────────────────────────────────────────────────────────────┐
│ [문제] 1번째 조건식이 T인 경우, 2번째가 T든 F든 결과가 T        │
│ [해결] 2번째 조건식에서 T와 F인 경우를 각각 하나씩 추가         │
└─────────────────────────────────────────────────────────────────┘

```

###   4. 기본 경로 테스트 (Basic Path Test)

**정의**: 원시 코드의 독립적인 경로가 최소한 한 번은 실행되는 테스트 케이스를 찾아 테스트

**목적**: 원시코드의 독립적인 경로를 모두 수행하는 것

**수행 절차**:

```text
┌──────┐    ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐    ┌──────┐
│ 코드  │───►│ ①순서도 │───►│ ②흐름   │───►│ ③순환   │───►│ ④독립적 │───►│ ⑤테스트│───► 테스트
│      │    │   작성  │    │  그래프  │    │복잡도    │    │ 경로   │     │ 케이스 │
│      │    │        │    │  작성   │    │ 계산     │    │ 정의   │     │ 작성  │
└──────┘    └────────┘    └────────┘    └────────┘    └────────┘    └──────┘

```

#### 단계 1: 순서도 작성

- 문장, 조건문(if-then-else), 반복문(for)의 3가지 기본 구조로 표현

![소프트웨어 테스팅 개요](/assets/images/posts/software-testing-overview/software-testing-overview-13.png)

#### 단계 2: 흐름 그래프 작성

**기본 구조**:

![소프트웨어 테스팅 개요](/assets/images/posts/software-testing-overview/software-testing-overview-14.png)

#### 단계 3: 순환 복잡도 계산

**순환 복잡도 (Cyclomatic Complexity, CC)**:
매케이브가 정의한 메트릭으로 원시 코드의 복잡도를 정량적으로 평가

**계산 공식**:

![소프트웨어 테스팅 개요](/assets/images/posts/software-testing-overview/software-testing-overview-15.png)

#### 단계 4: 독립적 경로 정의

순환 복잡도가 3이므로 독립적인 경로는 3개:

```text
• 경로 1: 1-2-3-8
• 경로 2: 1-2-4-5-6-8
• 경로 3: 1-2-4-5-7-8

```

#### 단계 5: 테스트 케이스 작성

3개의 경로에 해당하는 테스트 데이터를 생성

### 5. 검증 기준 비교 요약

```text
┌───────────────────┬─────────────────────────────────────────────────────┐
│     검증 기준                            특징
├───────────────────┼─────────────────────────────────────────────────────┤
│   문장 검증 기준        모든 문장 최소 1번 실행
│ (Statement)           → 가장 약함, 논리 연산자 오류 못 찾음
├───────────────────┼─────────────────────────────────────────────────────┤
│   분기 검증 기준        전체 조건문의 T/F 각각 1번씩
│ (Branch)          │   → 개별 조건식 오류 못 찾음
├───────────────────┼─────────────────────────────────────────────────────┤
│   조건 검증 기준        개별 조건식의 T/F 각각 1번씩
│ (Condition)           → 분기 검증 보완, 개별 오류 발견 가능
├───────────────────┼─────────────────────────────────────────────────────┤
│ 분기/조건 검증기준        개별 조건식 + 전체 조건문 모두 만족
│ (Branch/Condition)    → 마스크 문제 존재
├───────────────────┼─────────────────────────────────────────────────────┤
│ 다중 조건 검증기준        마스크 문제까지 해결
│ (Multiple Cond.)      → 가장 강력, 테스트 케이스 수 증가
├───────────────────┼─────────────────────────────────────────────────────┤
│  기본 경로 테스트        순환 복잡도 기반 독립 경로 테스트
│ (Basic Path)          → 정량적 복잡도 평가 가능
└───────────────────┴─────────────────────────────────────────────────────┘

```

**검증 강도 순서**:

```text
문장 검증 < 분기 검증 < 조건 검증 < 분기/조건 검증 < 다중 조건 검증
  (약함)                                      (강함)

```
