# Blog Post Writer (File) — 강의자료 파일 → Jekyll 블로그 포스트

## 개요
대학 강의자료 파일(PDF/PPT)을 받아서 Jekyll 블로그(l2juhan.github.io) 포스트로 변환하는 메인 스킬이다.
이미지는 **블로그 내용과의 관련성을 먼저 검증**한 뒤, 관련 있는 것만 `image-extractor` 서브 에이전트로 추출한다.

## 사용법
```
/blog-post-writer-file <카테고리>
```
- 카테고리를 인자로 받는다. `$ARGUMENTS`에 카테고리가 전달된다.
- 예시: `/blog-post-writer-file Operating-System`
- 카테고리 없이 호출하면 파일 내용을 분석하여 자동 결정한다.

---

## 핵심 원칙

### 1. 사람이 쓴 것처럼
- **AI 냄새 제거**: "~에 대해 알아보겠습니다", "이번 포스트에서는", "마무리하며" 같은 AI 틀에 박힌 표현 금지
- **~다 체 필수**: "~이다", "~한다", "~된다", "~있다" 등 딱딱하고 시크한 말투. 군더더기 없이 건조하게 서술한다
- **간결한 전환**: "그렇다면 왜 필요한가.", "여기서 핵심은", "하지만 이것만으로는 부족하다"
- **불필요한 인사/맺음 금지**: "안녕하세요", "감사합니다", "도움이 되셨길" 절대 사용하지 않는다

### 1-1. 콘텐츠 수준
- **컴퓨터공학과 학부생이 이해할 수 있는 수준**으로 작성한다
- 지나치게 쉽게 풀어쓰지 않는다. 전공 용어는 그대로 쓰되, 처음 등장 시 간결하게 정의한다
- 대학원 수준의 심화 내용은 다루지 않거나, 다루더라도 "심화" 등으로 별도 표기한다
- 수식이나 알고리즘은 교과서 수준의 표기를 따른다

### 2. 한국 CS 블로그 글 흐름
```
[도입부] 왜 이 개념이 필요한지 1-2문장 동기부여
    ↓
[핵심 개념] 정의 + 직관적 설명 (비유/예시)
    ↓
[구조/원리] 이미지와 함께 동작 방식 설명
    ↓
[비교/분류] 관련 개념 비교 (표 활용)
    ↓
[심화/주의점] 실무/시험 포인트
    ↓
(선택) [정리] 핵심만 짧게
```

**좋은 도입부:** "메모리는 빠를수록 비싸고, 클수록 느리다. 이 딜레마를 해결하기 위해 계층적 구조가 필요하다."
**나쁜 도입부 (금지):** "이번 포스트에서는 메모리 계층 구조에 대해 알아보겠습니다."

## 워크플로우

### Step 1: 파일 내용 분석

1. `$ARGUMENTS`를 확인하여 카테고리가 지정되었는지 파악한다
2. 파일의 텍스트 내용을 전부 읽는다
3. 전체 구조를 파악한다:
   - 다루는 주제/개념 목록
   - 각 개념의 깊이와 범위
4. 블로그 포스트 아웃라인을 설계한다:
   - 제목, 태그
   - 카테고리: `$ARGUMENTS`에서 받았으면 그대로 사용, 없으면 내용 기반으로 결정
   - 섹션 구성 및 흐름

### Step 2: 이미지 관련성 판단 (핵심 단계)

파일 내 이미지를 **블로그 내용과 대조하여 관련성을 검증**한다.
관련 있는 이미지만 추출 대상으로 선별하고, 나머지는 제외한다.

**판단 기준:**

| 기준 | 관련 있음 (추출) | 관련 없음 (제외) |
|------|-----------------|-----------------|
| 개념 설명 보조 | 다이어그램, 구조도, 플로우차트 | 장식용 클립아트 |
| 본문 직접 참조 | 본문에서 "위 그림과 같이" 등으로 참조 | 본문과 무관한 삽화 |
| 이해도 향상 | 텍스트만으로 설명하기 어려운 시각 자료 | 텍스트로 충분히 대체 가능 |
| 유일성 | 고유한 정보를 담는 이미지 | 비슷한 내용의 중복 이미지 |

**관련성 검증 프로세스:**

```
1. 파일에서 이미지가 있는 페이지 목록 확인
2. 각 페이지의 이미지에 대해:
   a. 이미지 주변 텍스트 컨텍스트 분석
   b. 블로그 글의 어떤 섹션에서 이 이미지가 필요한지 판단
   c. "이 이미지 없이 해당 섹션을 설명할 수 있는가?" 자문
   d. 관련성 판정: RELEVANT / IRRELEVANT
   e. RELEVANT인 경우, 추출 모드 결정:
      - extract: 이미지 단독으로 의미가 완성되는 경우
      - crop: 이미지 + 주변 텍스트 라벨이 세트인 경우
3. RELEVANT 판정된 이미지만 추출 요청 목록에 추가
```

**모드 판단 기준:**
- `extract`: 사진, 독립 다이어그램, 로고, 차트 등 이미지 자체로 내용이 완결되는 경우
- `crop`: 메모리 레이아웃 + 주소 라벨, 회로도 + 신호명, 그래프 + 축 수식 등 주변 텍스트가 이미지와 한 세트인 경우

**판단 결과 형식 (내부용):**
```
이미지 관련성 판단:
  - p.3 다이어그램: RELEVANT (extract) → "메모리 계층 구조" 섹션, 이미지 단독으로 완결
  - p.4 표 캡처: IRRELEVANT → 마크다운 표로 대체 가능
  - p.5 메모리 레이아웃: RELEVANT (crop) → 주소 라벨($sp, $gp)이 이미지와 세트
  - p.1 대학 로고: IRRELEVANT → 장식용
  - p.8 코드 스크린샷: IRRELEVANT → 코드 블록으로 직접 작성 가능
```

### Step 3: 이미지 추출 — image-extractor 서브 에이전트

Step 2에서 RELEVANT로 판정된 이미지가 **1개 이상**인 경우에만 `image-extractor` 스킬을 호출한다.

**호출 시 전달 정보:**
- 파일 경로
- 출력 디렉토리: 프로젝트 내 `assets/images/posts/[slug-name]/`
- 슬러그: 포스트 파일명과 동일
- RELEVANT 이미지 목록 (페이지 번호 + 이미지 설명 + 모드)

**전달 예시:**
```
추출 요청:
  - 페이지 3: 메모리 계층 피라미드 (mode: extract)
  - 페이지 5: MIPS 메모리 레이아웃 + 주소 라벨 (mode: crop)
  - 페이지 8: 캐시 매핑 구조도 (mode: extract)
```

**RELEVANT 이미지가 0개인 경우:**
- image-extractor를 호출하지 않고 Step 4로 바로 진행
- 코드 블록, 표, 수식 등으로 시각적 요소 보충

### Step 4: 블로그 포스트 마크다운 작성

#### Frontmatter
```yaml
---
layout: post
title: "제목 — 부제목"
date: YYYY-MM-DD
categories: [카테고리]
tags: [tag1, tag2, tag3]
description: "한 줄 설명"
---
```

**카테고리** (`$ARGUMENTS`로 전달받음):

기존 카테고리 목록:
- `Cloud` / `Database` / `Dev-Tools` / `Frontend` / `Linux`
- `Operating-System` / `Software-Engineering`
- `Computer-Architecture` / `Algorithm` / `Network`

카테고리 처리 로직:
1. `$ARGUMENTS`에 카테고리가 있으면 해당 값을 사용한다
2. 없으면 파일 내용을 분석하여 기존 카테고리 중 가장 적합한 것을 선택한다
3. 전달받은 카테고리가 기존 목록에 없으면 **새 카테고리로 생성**한다:
   - frontmatter의 `categories`에 그대로 사용한다
   - Jekyll은 categories 값을 기반으로 자동 라우팅하므로 별도 설정 불필요하다
   - 단, 카테고리명은 기존 네이밍 컨벤션을 따른다 (PascalCase 또는 kebab-case, 예: `Computer-Architecture`)

**제목 규칙:**
- 영어 주제명 있으면: `"영어 제목 — 한국어 부제"`
- 한국어만: `"한국어 제목"`
- 기존 포스트 스타일과 일관성 유지

#### 본문

**구조:**
```markdown
# 메인 제목

한 줄 설명

## 첫 번째 개념

[동기부여 1-2문장]

[핵심 설명]

![이미지 설명](/assets/images/posts/[slug]/[slug]-N.png)

[이미지 참조 설명]

## 두 번째 개념
...
```

**문체:**
- ~다 체: "~이다", "~한다", "~된다", "~하는 구조이다"
- 설명: "즉, ~인 셈이다", "핵심은 ~라는 점이다"
- 전환: "그렇다면 왜 필요한가.", "여기서 문제가 발생한다", "하지만 이것만으로는 부족하다"
- 톤: 딱딱하고 시크하게. 감정 표현이나 감탄 없이 사실 위주로 서술한다
- 강조: **볼드** 활용
- 코드: 인라인 `` ` ` ``, 블록 ` ``` `
- 용어: 한국어(영어) 병기. 예: "교착상태(Deadlock)"

**이미지 경로:**
```markdown
![간결한 설명](/assets/images/posts/[slug-name]/[slug-name]-N.png)
```
- N은 image-extractor가 부여한 순번과 일치
- 이미지 바로 위/아래에서 자연스럽게 참조

**금지 표현:**
- "~에 대해 알아보겠습니다" / "~를 살펴보겠습니다"
- "이번 포스트에서는" / "다음 포스트에서"
- "안녕하세요" / "감사합니다" / "도움이 되셨길"
- "자, 그럼" / "그럼 바로"
- 음슴체 ("~임", "~됨", "~함") 사용 금지
- 이모지
- 같은 내용 반복 / 지나치게 긴 서론

### Step 5: 결과물 전달

1. `_posts/YYYY-MM-DD-slug-name.md` 생성
2. (이미지 있으면) `assets/images/posts/[slug-name]/` 폴더에 이미지 배치
3. 사용자에게 간결하게 안내

## 파일 구조
```
블로그 레포/
├── _posts/
│   └── YYYY-MM-DD-slug-name.md
└── assets/images/posts/[slug-name]/
    ├── [slug-name]-1.png
    ├── [slug-name]-2.png
    └── ...
```

## 글 길이 가이드
- 짧은 주제: 200~400줄
- 일반 강의 1회분: 300~600줄
- 큰 주제: 400~800줄
- 너무 길면 분할 제안
